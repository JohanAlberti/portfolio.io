Study of the LoRa protocol
Context

For the course on protocols for connected objects I had to realize with Romain Saboret a presentation summarizing a documentary research on the LoRa protocol.

Resolution of problem

With my partner I had to define the main characteristics of the protocol in order to compare it properly with its competitors. LoRa bases its communication on :

  • Emerging communication technology
  • Long Range
  • Large-scale network coverage

We have therefore chosen to focus our comparisons on :

  • Mobility
  • Coverage
  • Cost
  • Consumption
  • Data rate
  • Number of Devices

The presentation revolves around the physical layer LoRa and LoRaWAN.

Introduction to Software-Defined-Radio (GNU Radio)
Context

During this team project, I focused on the reception of real communication signals (e.g. broadcasting, aeronautical communications, etc.). Thus a specific part of the wide frequency range of radio spectrum is allocated to each signal with regards to the targeted applications and constraints.

Resolution of problem

Firstly, in this report, we demonstrate that, in the case of narrowband signals, the IQ transceivers (In-phase/Quadrature) allow transmissions with frequency transportation without altering data. Secondly, we report our work on FM broadcasting, where our aim was to demodulate it in real time. Thirdly, the VOLMET AM broadcasting have been treated, again in real time.

1. Know how to position the main standards of the Internet of Things

Most standards (Auto-ID Labs, FDA…) are limited to specific uses, e.g. FDA allows the identification of medical components. It is therefore necessary to define an open and deployable standard for all connected objects.

The oneM2M standard is a project founded in 2013 by various standardization bodies specialized in information and communication technologies (ARIB, ATIS, ETSI, etc.) [2]. It is positioned as an open standard for horizontal platforms. Its structure is composed of an application layer, a service layer and a network layer, which allows for a high level of interoperability. Thus, it is useful for many IoT applications (connected houses, smart cities, transport, health…) [3].

Other standards such as IIC (Industrial Internet Consortium), which focuses rather on the industrial aspect of IoT and appeared after oneM2M but offers a complete architecture with interoperability in many industrial domains (Industry 4.0).

2. Deploy a standard-compliant architecture and set up a sensor network system for services

2.1. Deploying and configuring an IoT architecture using OM2M

The project aims at deploying an IoT architecture on a Phillips HUE lamp using the oneM2M architecture thanks to OM2M eclipse. The lamp communicates to a gateway using ZigBee technology.

The objective is to provide an API following the REST architecture to the gateway. It is then necessary to create an inter functional proxy entity allowing the connection between the HUE gateway and oneM2M.

Figure 2 – Interacting Elements

If we translate all of the above into the oneM2M standard, we obtain an architecture based on 3 different nodes linking the interacting entities:

  • An Infrastructure Node (IN): the OM2M server / router 192.168.X.1
  • A Middle Node (MN): the HUE Bridge gateway
  • A Dedicated Node (DNA) Application: the Phillips HUE lamp

The lamp is controlled through a Dashboard type user interface that allows to communicate with OM2M and also to receive its status (brightness, state and hue).

Figure 3 – Architecture to be deployed

So we end up with an application layer that includes the lamp, the HUE Bridge gateway and the router. The services are provided by the mn-cse and the in-cse. Finally, the transport layer intervenes between the User Interface and OM2M through a REST API.

2.2 Interacting with objects using a REST architecture

To interact with the connected objects, oneM2M resources must be used. After creating the AE corresponding to our LAMP_1, we placed a CNT (container) of type DATA. When the lamp changes state, the REST API sends an HTTP POST request and creates a new CIN within the DATA CNT, which contains the new lamp data.

Figure 4 – Structure of an AE in OM2M

By performing a SUB (subscription) within the CNT DATA, the user can be notified as soon as a new CIN is received. Thus, by specifying an access point (poa) designating a server (local in our case), we send to the user the status of the lamp.

In the same way, the user can control the lamp by posting a CIN with the values of his choice through an HTTP POST request via the REST API.

2.3 Integrating new/other technology into an IoT architecture

The principle of Interworking Proxy Entity of oneM2M is, by means of an SDK, to make it possible to access values from Philips devices and to monitor them directly from OM2M.

Philips provides a complete SDK allowing local control of the lamp (from the HUE Bridge gateway). We had to adapt our OM2M so that it can receive and send the information in the right format to the lamp.

To do so, we had to translate between the OM2M HTTP request and the functions provided by the corresponding SDK to access the lamp and retrieve its status.

3. Deploy a composite application between various technologies using NODE-RED based on a standardised middleware

Node-RED is a graphical development tool dedicated to interconnect hardware with APIs and online services. The editor is entirely web-based and is used to create javascript functions in the form of nodes in order to link them together to form flows.

Node-RED provides advantages for not building interconnections yourself using nodes created by other users that can be available to all. Thus, a library of basic nodes is provided at node-RED startup. Likewise, graphically as well as in command lines, it is easily possible to include other nodes like those of OM2M.

We can thus easily create different objects in OM2M like AEs, containers or content instances.

Figure 5 – Node RED : OM2M

Figure 6 – Node RED : Initialization and  « get state »

The REST application allows to send to the OM2M server the values of the Philips Hue lamp such as brightness, state or hue. With node-RED and OM2M nodes, we can retrieve this information and display it in a debug window, or even in html thanks to the mustache template allowing to switch from JSON input to html. Another possibility is to use another library to directly create a dashboard.

Figure 7 – Node RED : Dashboard

We can send commands to the OM2M server and retrieve information from the lamp. We can then create conditions allowing us to change some values of the lamp if it does not follow selected criteria. In this example, we have chosen to look at the state and brightness of the lamp and to increase its brightness if it is too low.

Figure 8 – Node RED : « change state »

By retrieving the time with the timestamp node, it is possible to create an automatic command increasing the lamp after 17:00 if it is too low. Almost all conditions are possible thanks to the function block where scripts can be written to modify the message according to what it contains.

Figure 9 – Node RED : « change state » at 5 pm

It is also possible to create subscriptions and manage requests sent by the OM2M server when new data arrives. By combining these nodes with those seen previously, it is possible to create conditions that are tested at each new state of the lamp and to activate or not a command.

Figure 10 – Node RED : Subscription

Thanks to the nodes of the dashboard library, we were able to connect the OM2M tools seen previously to a visual rendering provided. Inputs and outputs are managed independently, this way, if changes are made by a third party application or by manually changing the lamp status, the dashboard is updated automatically.

Figure 11 – RED Node: Dashboard Update

Figure 12 – UI from a browser

The dashboard then adapts to all screen formats. We can now access the dashboard’s corresponding web page with a mobile phone, where it is now possible to view the status and characteristics of the lamp, and to change these values simply.

By publishing the OM2M server and node-RED on the internet and having an internet connection for sending and receiving lamp data, it is possible to change the status of the lamp or view its values from anywhere in the world.
